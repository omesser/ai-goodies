## What this project is

A field data collection system for partial discharge (PD) experiments on power towers. Field operators use a phone app to log experiments; the data flows through AWS into a MySQL truth table consumed by Airflow pipelines and displayed in Grafana.

---

## Active architecture vs legacy code

**The active production stack is CloudFront → API Gateway → SQS → Lambda → MySQL → Airflow.**

There is also a legacy FastAPI + PostgreSQL + Docker stack in `api/`, `web/`, `docker-compose*.yml`. This code still exists and runs for local development, but it is not the production system. Do not:
- Document the legacy stack as the primary architecture
- Extend the legacy stack for new production features
- Suggest deploying the old EC2/Docker setup as a solution

When a user asks about "the architecture", they mean the CDK stack in `cdk/`. When they ask about local dev, the Docker stack is fine.

## Key facts about the system

**Truth table** — `partial_discharge_truth_table` in MySQL (at `prisma_projects @ annotations.prismaphotonics.net`) is the single source of truth consumed by all downstream processes. Every experiment eventually lands here.

**File timestamps** — Device filenames use `YYYYMMDD_HHMMSS_*` format. Timestamps are in `Asia/Jerusalem` timezone by default. Always convert to UTC before matching against `start_time`/`end_time` (which are UTC). The matching window is ±15 minutes.

**Idempotency** — `client_request_id` (UUID, generated by the phone app) is the idempotency key for experiment submission. API Gateway and SQS can deliver duplicates; downstream code must handle this.

**Credentials** — Phones get temporary AWS credentials from Cognito Identity Pool (unauthenticated). These sign API Gateway requests via SigV4. The Cognito unauth role must have `execute-api:Invoke` and `sqs:SendMessage`. No IAM user credentials go to phones.

**field.html** — The PWA source lives at `web/public/field.html`. CloudFront serves `web/public/` from S3 — deploy with `deploy_field_app.bat` or `deploy_field_app.py`.

**Lambda** — `lambda_functions/phone_ingestion/dist/handler.py` is the deployed artifact. The `dist/` directory is what gets zipped and deployed, not the source root.

## Infrastructure

CDK stack is in `cdk/`. It is a Python CDK app.

```
cdk/
  phone_pipeline_stack.py   # PhonePipelineStack — SQS, Cognito, API GW, S3, CloudFront
  app.py                    # Stack instantiation; reads env vars for account/region/vpc
```

To deploy:
```bash
cd cdk
pip install -r requirements.txt
cdk deploy
```

Required env vars for CDK: `AWS_ACCOUNT_ID`, `AWS_REGION`, optionally `VPC_ID` and `COGNITO_UNAUTH_ROLE_ARN`.

After deploy, attach the Cognito role to the identity pool manually:
```bash
aws cognito-identity set-identity-pool-roles \
  --identity-pool-id <IdentityPoolId> \
  --roles unauthenticated=<COGNITO_UNAUTH_ROLE_ARN>
```

## Workflow preferences

**Confirm before destructive operations.** This includes: deleting files, force-pushing, dropping DB tables, modifying shared infrastructure, or overwriting uncommitted work. Ask first.

**Don't over-engineer.** A bug fix doesn't need surrounding cleanup. A one-shot script doesn't need a class hierarchy. Three similar lines is better than a premature abstraction. Build only what the task requires.

**Don't add unnecessary comments.** Only comment when the *why* is non-obvious — a hidden constraint, a subtle invariant, a workaround for a specific bug. Don't narrate what the code does.

**Prefer editing existing files** over creating new ones.

**Keep responses concise.** State what changed and what's next. Don't summarize the diff back at the end of a response.

**Docs belong in README.md or docs/.** Do not create intermediate analysis files unless explicitly asked.
